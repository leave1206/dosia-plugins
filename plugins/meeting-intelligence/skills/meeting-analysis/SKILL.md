## Step 0: 会议画像

读取逐字稿后，先建立会议画像（输出到会议记录开头）：

### 0.1 会议类型识别

从以下类型中选择（可多选）：

| 类型 | 识别信号 | 示例 |
|------|---------|------|
| **战略决策会** | 方法论输出、战略方向、价值观讨论 | CEO 讲 AI 内容体系 |
| **项目推进会** | 进度汇报、资源协调、排期讨论 | 投流数据复盘+下步计划 |
| **客户对接会** | 有外部参与者、需求对接、方案汇报 | 品牌方 Q2 策略对齐 |
| **项目复盘会** | 回顾成果、分析问题、总结经验 | 某活动结束后复盘 |
| **团队周会/日会** | 定期、多人轮流汇报、问题收集 | 内容团队周进度 |
| **1on1/指导会** | 上级对下级、培训、辅导 | 组长带新人过流程 |
| **头脑风暴/讨论会** | 无明确议程、发散讨论、方案探讨 | 几个策划讨论新玩法 |
| **跨部门协作会** | 多部门、接口对齐、职责划分 | 投流+内容+策划三方对齐 |

### 0.2 参会人权限层级

识别每个参会人的层级（影响分析策略）：

| 层级 | 角色示例 | 在分析中的作用 |
|------|---------|--------------|
| **L1 决策层** | CEO、创始人、合伙人 | 其观点可作为知识沉淀的来源 |
| **L2 管理层** | 总监、组长、项目负责人 | 其观点可提议沉淀，需 L1 审批 |
| **L3 执行层** | AE、AM、运营、设计、剪辑 | 主要作为审查对象 + 问题/经验提取 |
| **外部** | 客户、供应商、合作方 | 信号捕捉对象，不做审查 |

### 0.3 飞书身份预解析

**在分析开始前建立姓名 → open_id 映射表，供 TODO 派发步骤使用。**

#### 数据来源优先级

1. **企业索引**（首选，O(1) 本地读取）

   ```
   路径: /Volumes/dosia-enterprise/enterprise/lark-directory.json
   结构: { "nasUser": { "name": "张三", "openId": "ou_xxx", ... } }
   ```

   用 Read 工具读取后，按 `name` 字段反查，建立映射表。

2. **飞书通讯录实时搜索**（兜底，仅当索引中找不到时）

   ```bash
   lark-cli contact +search-user --query "<姓名>" --format json --as user
   ```

   - 返回 1 条 → 使用该 `open_id`
   - 返回 0 条 → 标记为"未找到"
   - 返回多条（重名）→ 消歧义：优先取企业索引中存在的那条；若均不在索引中，列出候选姓名+部门，不自动选择

#### 输出：参会人身份表（贯穿后续步骤使用）

| 姓名 | 层级 | 飞书 open_id | 来源 | 状态 |
|------|------|-------------|------|------|
| 张三 | L2 | ou_xxx | 企业索引 | ✓ 已解析 |
| 李四 | L3 | ou_yyy | 通讯录搜索 | ✓ 已解析 |
| 王五 | L3 | — | 通讯录搜索 | ✗ 未找到 |
| 赵六 | 外部 | — | — | 跳过 |

> 外部参与者（客户/供应商）不做飞书解析，直接标注"外部"跳过。
> 个人模式（无 NAS）跳过企业索引，直接走通讯录搜索。

### 0.4 会议画像输出格式

```
会议类型: [类型1] + [类型2]（如有多个）
```

## Step 2: 分析策略路由

基于会议画像，决定执行哪些分析模块：

```
              ┌─────────────────────────────────────────┐
              │           所有会议都执行                  │
              │  [基础分析] + [TODO提取] + [问题发现]     │
              └────────────────┬────────────────────────┘
                               │
          ┌────────────────────┼────────────────────────┐
          │                    │                        │
    有 L1/L2 输出       有外部参与者              纯内部执行
    方法论/价值观       (客户/供应商)             (L2/L3 之间)
          │                    │                        │
    ┌─────▼─────┐      ┌──────▼──────┐         ┌───────▼──────┐
    │ 知识沉淀  │      │ 外部信号    │         │ 内部行为     │
    │ (提案制)  │      │ 捕捉+审查   │         │ 审查         │
    └───────────┘      └─────────────┘         └──────────────┘
```

**执行规则**：
- **知识沉淀**：当 L1 或 L2 在输出方法论、价值观、标准时触发
- **外部信号捕捉**：当有客户/供应商等外部参与者时触发
- **行为审查**：仅在 **无 L1 参与** 的会议中执行（L1 在场时由其直接指导，无需 Agent 审查）
- **提案生成**：知识沉淀发现的新内容，一律生成提案而非直接写入

## Step 3: 基础分析（所有会议）

### 3.1 参会人识别与角色判断

- 通过语言特征判断层级和角色
- 提取每个人的核心观点和立场
- 标注我方/外部

### 3.2 议题与决策提取

- 识别所有讨论主题
- 标注状态：已决策 / 待讨论 / 待确认 / 被搁置
- 记录每个决策的提出者和理由

### 3.3 TODO 提取与飞书派发

#### 提取阶段

- 明确的任务分配（含负责人、标准、截止）
- 隐含的待办（从讨论中推断但未明确分配）
- 上次会议遗留项的跟进状态（如可判断）

每条 TODO 格式：
```
- [ ] [任务描述] — 负责人: 张三 | 截止: YYYY-MM-DD | 标准: [验收条件]
```

#### 派发阶段（提取完成后立即执行）

对每条有明确负责人的 TODO，查 Step 0.3 的参会人身份表：

**有 open_id（已关联飞书）→ 直接创建飞书任务**

```bash
# 参考 lark-task skill 的 +create 用法
lark-cli task +create \
  --title "<任务描述>" \
  --due "<YYYY-MM-DD>" \
  --members "ou_xxx" \
  --as user
```

- `--members` 填该负责人的 `open_id`
- `--due` 有截止日期时填写，无则省略
- 创建成功后记录返回的任务链接

**无 open_id（未关联飞书）→ 仅记录，不发送**

标注为"未派发（未关联飞书）"，在会议记录的 TODO 章节注明。

**汇总发给当前用户（必须执行）**

通过飞书 IM 将汇总信息发送给触发本次分析的用户（即 system prompt `## 当前用户` 中的人）。

当前用户的 `open_id` 从 Step 0.3 参会人身份表中读取；若当前用户不在参会人列表中，从 `lark-directory.json` 按姓名查找。

```bash
# 向当前用户发送飞书私信（bot 身份，lark-im +messages-send）
lark-cli im +messages-send \
  --user-id "<当前用户 open_id>" \
  --as bot \
  --markdown "**📋 会议纪要已生成：<会议标题>**

**📄 纪要文档：** <会议记录文件路径>

**✅ 会后 TODO（共 N 条）：**
- 张三：完成投流方案（截止 MM-DD）
- 李四：更新数据报表
- 王五：整理客户反馈（⚠️ 未关联飞书，需手动通知）"
```

- `--as bot` 为飞书 IM 发送的唯一身份，需确保 bot 已与该用户有会话关系（首次可能需要用户先主动添加 bot）
- `--markdown` 支持加粗、列表，比 `--text` 更清晰
- 若当前用户 open_id 无法解析（未关联飞书），跳过此步，仅在对话中输出汇总
- 消息内容：纪要文件路径 + 全部 TODO 列表（含未关联飞书的成员，注明需手动跟进）

#### 派发结果汇总（输出到会议记录 § 七）

```
飞书派发结果：

个人派发（发给各自负责人）：
✓ 张三: "完成投流方案" → https://applink.feishu.cn/...
✓ 李四: "更新数据报表" → https://applink.feishu.cn/...
⊘ 王五: "整理客户反馈" → 未派发（未关联飞书）
— 赵六: 外部人员，跳过

汇总通知（发给当前用户 <姓名>）：
✓ 飞书消息已发送 → 含纪要路径 + 全部 N 条 TODO
```

### 3.4 问题发现

- 流程/协作中暴露的问题
- 资源/工具/能力缺口
- 信息不对称或沟通断裂
- 决策被搁置或反复讨论无结论的议题

## Step 4: 知识沉淀（有 L1/L2 方法论输出时）

### 4.1 知识提取

- 业务思考与洞察（标注成熟度 + 提出者层级）
- 可结构化的方法论、框架、模型
- 新的 SOP/流程/标准
- 对已有价值观/SOP 的强化、修正或补充

### 4.2 生成变更提案（PROPOSAL）

**所有对 foundations/ 的变更一律以提案形式输出，不直接修改文件。**

**PROPOSAL 必须去项目化**：提案内容不得包含具体客户名、产品名、项目名称或专属数据。提案应抽象为可复用于同类项目的通用方法论，具体项目数据只能作为"变更理由"中的验证来源引用，不进入正文。

提案格式：

```markdown
## PROPOSAL-[序号]: [变更标题]

- 来源: [会议名称] [提出者(层级)]
- 目标文档: foundations/[文件名].md
- 变更类型: 新增 / 修改 / 删除 / 升级成熟度
- 提案权限: L1直接批准 / L2提议需L1确认

### 当前内容（如修改/删除）
> [引用现有文档中的内容]

### 提议变更
[去项目化的通用方法论内容，不含具体客户/产品/数据]

### 变更理由
[为什么要做这个变更；可引用原文和项目数据作为验证来源]

### 审批状态
- [ ] 待审批
```

### 4.3 提案权限规则

| 提出者层级 | 提案处理 |
|-----------|---------|
| **L1 (CEO/创始人)** | 自动标记为 `推荐批准`，仍需用户确认后才写入 |
| **L2 (组长/总监)** | 标记为 `待 L1 审批`，建议在下次 L1 会议中确认 |
| **L3 (执行层)** | 标记为 `待评估`，需 L2+ 确认是否有沉淀价值 |
| **会议共识** | 如果多人达成一致，标记 `会议共识-推荐批准` |

### 4.4 提案批量处理

在会议记录末尾汇总所有提案，用户可以：
- `批准 PROPOSAL-1,3,5` → Agent 执行写入
- `拒绝 PROPOSAL-2` → 标记已拒绝
- `修改 PROPOSAL-4: [修改意见]` → Agent 调整后重新提交
- `全部批准` → 一次性写入所有提案

## Step 5: 行为审查（所有会议，强度自适应）

### 5.1 审查触发条件与强度

**前提：仅在无 L1 参与的会议中执行行为审查。** L1 在场时由其直接指导，Agent 不越位审查。

| 会议类型（无 L1 时） | 审查强度 | 说明 |
|---------------------|---------|------|
| 客户对接会 | **全面审查** | 逐条对照价值观+SOP，输出完整风险报告 |
| 项目推进会 | **重点审查** | 聚焦执行力、SOP 遵守、沟通质量 |
| 项目复盘会 | **回顾审查** | 对照目标 vs 结果，提取经验教训 |
| 1on1/指导会（L2→L3） | **轻度审查** | 关注 L2 是否传达了正确的价值观 |
| 团队周会 | **轻度审查** | 关注效率、是否有推诿/甩锅行为 |
| 头脑风暴 | **不审查** | 只提取有价值的想法和洞察 |

**有 L1 参与时**：跳过整个 Step 5，仅执行知识沉淀。

### 5.2 审查维度

对照 foundations/ 中所有可适用的标准：

**A. 价值观维度**（对照 company-values.md）

逐条扫描，针对会议中可观察到的行为给出 RED/YELLOW/GREEN 评级：

| 评级 | 定义 |
|------|------|
| RED | 明确违反红线或核心价值观 |
| YELLOW | 未达 80 分标准，存在改进空间 |
| GREEN | 符合标准 |
| N/A | 该条在本次会议中无法观察/不适用 |

**B. SOP 维度**（对照各 SOP 文档）

- 该会议涉及的工作流程是否按 SOP 执行
- 有无跳步、遗漏、顺序错误
- AE/AM 服务评分（如适用）

**C. 协作维度**（通用，不依赖特定文档）

- 沟通效率：是否有反复解释/理解偏差
- 责任意识：是否有推诿/甩锅行为
- 主动性：是否只回应不主动
- 信息完整度：汇报/交付是否充分

### 5.3 风险点报告

对每个 RED/YELLOW 项输出：

```
[RED/YELLOW] 风险点 #N: [简短标题]
- 原文引用: "[具体对话内容]"
- 涉及人员: [姓名/代号(层级)]
- 对标标准: [foundations/xxx.md → 第X章 → 具体条目]
- 影响分析: [不纠正会导致什么]
- 改进建议: [具体应该怎么做]
- 话术示例: "[可直接使用的改进话术]"
```

### 5.4 亮点识别

不只找问题，也要发现做得好的地方：
- 哪些行为符合甚至超越了价值观标准
- 哪些人展现了 90 分水平的表现
- 可作为团队正面案例分享

## Step 6: 外部信号捕捉（有外部参与者时）

### 6.1 客户/外部方信号分析

| 信号类型 | 识别方法 | 输出 |
|---------|---------|------|
| 满意信号 | 正面评价、主动提议扩大合作 | 记录并标注 |
| 不满信号 | 语气变化、反复追问、沉默、直接批评 | 标注 + 建议应对 |
| 潜在需求 | 话里有话、暗示、旁敲侧击 | 推断 + 建议行动 |
| 决策信号 | 表态、承诺、拒绝 | 记录并跟进 |

### 6.2 关系温度评估

```
强信任: 主动分享信息、给建议、推荐资源
中性:   正常业务对接，不冷不热
弱信任: 指令式沟通、不分享背景、频繁检查
危机:   明确批评、威胁更换、减少沟通
```

基于信号分析，给出下一步行动建议（参照"要一给三"等价值观）。

## Step 8: 双视角点评（所有会议的最终章节）

**此章节包含两个独立的声音，各有不同的知识边界和思维方式。始终输出，不裁剪。**

### 8.1 两个声音的定义

| | CEO 替身 | 行业专家视角 |
|--|---------|-----------|
| **身份** | 深谙 foundations/ 全部内容的公司最高决策者 | 融合五个领域的资深专家：消费品 + 互联网 + 内容营销 + AI 应用 + AI Agent 时代的组织管理 |
| **知识边界** | 严格锚定 foundations/ 中的条目 | 消费品牌方法论、平台生态、内容营销前沿、AI 落地实践、人机协作组织设计 |
| **思维特征** | 内部视角：我的公司、我的团队、我的标准 | 行业视角：见过足够多的公司和案例，能看到当局者看不到的模式 |
| **表达风格** | 直接、高压、落地（模拟 CEO 口吻） | 洞察驱动、举一反三、用行业案例佐证、点到本质 |
| **批评方式** | "你这个做法不对，应该这么做" | "这个做法在XX行业/公司曾经导致了YY，根本原因是ZZ" |
| **可以做** | 批评、表扬、下指令、设截止时间 | 提供行业对标、指出结构性问题、预判趋势、建议实验、评估组织形态 |
| **不可以做** | 不能超越 foundations/ 创造新标准 | 不能下指令（只能建议），不能否定 foundations/ 中的 [CORE] 项 |

### 8.2 输出结构

```markdown

### Part B: 行业专家视角

> 以一位在消费品、互联网、内容营销、AI 四个领域融会贯通的资深行业专家身份，
> 提供跳出公司内部框架的洞察。以下观点可能与 CEO 视角互补、延伸、甚至存在张力
```

### 8.3 CEO 替身写作原则

| 原则 | 说明 |
|------|------|
| **锚定 foundations/** | 批评和建议必须引用具体条目 |
| **模拟 CEO 口吻** | 直接、不绕弯、有高度但落地 |
| **不重复前文** | 聚焦更高维度，不复述审查报告 |
| **指示可执行** | 不说"加强沟通"，说"明天前给XX发消息，内容包含XX" |

**CEO 口吻校准**（从已有会议提取）：
- 直接指出问题并给路径（"你这个思维方式绝对不对的...应该是..."）
- 用类比说明（"你哪有 LV 会这么讲自己的"）
- 强调本质（"本质上就是处理弱信任关系的能力不足"）
- 评分框架（"60 分是...90 分是..."）
- 落到行动（"今天下班前搞定"）

**L1 在场时**：转为"回看与补充"——补充 CEO 可能遗漏的点、评估指示可行性、判断团队理解程度。

### 8.4 行业专家视角写作原则

| 原则 | 说明 |
|------|------|
| **行业案例佐证** | 引用具体的行业案例/公司实践（"完美日记在同阶段..."、"蕉内的内容团队..."） |
| **五域融合思考** | 每个洞察尽量连接消费品+互联网+内容营销+AI应用+AI时代组织管理中的至少两个领域 |
| **点到本质** | 不停留在现象描述，直接给出"这背后的结构性原因是..." |
| **承认不确定性** | 不确定的观点用"可能"、"值得验证"、"一个假设是" |
| **与 CEO 视角可以有张力** | 这是特性——提供决策者未考虑的角度 |
| **不否定 [CORE] 标准** | 可以建议讨论或提供反例，但不直接说"这条标准是错的" |
| **建议而非指令** | 行业专家是顾问角色，提供洞察和选项，决策权在公司 |
| **控制篇幅** | 没有深刻洞察时写"本次会议无需额外补充"，不注水 |

## Step 9: 提案处理

> **角色判断**：从 system prompt「当前用户」部分读取角色信息，决定审批路径。

### 路径 A: 管理员（角色 = 管理员 / admin）

管理员走 **inline 审批**，在当前对话中直接处理。

生成记录后，如果有 PROPOSAL，提示用户：

```
本次会议生成了 N 个 foundations 变更提案：
- PROPOSAL-1: [标题] (推荐批准/待L1审批/待评估)
- PROPOSAL-2: [标题] (推荐批准/待L1审批/待评估)
...

请回复操作：
- "批准 1,2,3" 或 "全部批准" → 写入 foundations/
- "拒绝 2" → 标记已拒绝
- "修改 3: [意见]" → 调整后重新提交
- "暂不处理" → 提案保留在会议记录中，后续可回溯
```

用户批准后，才执行对 foundations/ 的实际修改，并在文档变更日志中记录。

> 提案审批流程由 `meeting-intelligence:proposal-workflow` skill 处理。用户回复批准/拒绝/修改指令后，调用该 skill 执行写入，不要在 meeting-analysis skill 内直接修改文件。

### 路径 B: 非管理员（所有其他角色）

非管理员 **无权直接写入 foundations/**，提案自动提交管理员审批。

1. **提取所有 PROPOSAL**，写入审批队列文件：

   文件路径: `inc-knowledge/proposals/pending/{YYYY-MM-DD}-{HHMM}-{用户名}-{会议概述}.md`

   文件格式：
   ```markdown
   ---
   提交人: {当前用户名}
   提交时间: {ISO 时间戳}
   会议记录: meetings/{对应会议记录文件名}
   提案数量: {N}
   状态: 待审批
   ---

   ## PROPOSAL-1: {标题}

   - 目标文档: foundations/{路径}
   - 变更类型: 新增 / 修改 / 删除 / 升级成熟度
   - 来源: {发言人}({层级})

   ### 当前内容（如修改/删除）
   > {引用现有内容}

   ### 提议内容
   {去项目化的通用方法论内容}

   ### 变更理由
   {为什么做这个变更}

   ---

   ## PROPOSAL-2: {标题}
   ...
   ```

2. **回复用户确认**：

   ```
   📋 本次会议生成了 N 个知识库变更提案：
   - PROPOSAL-1: {标题} → foundations/{路径}
   - PROPOSAL-2: {标题} → foundations/{路径}
   ...

   ✅ 会议记录已保存: meetings/{文件名}
   📤 变更提案已提交管理员审批: proposals/pending/{文件名}
   ```

3. **不提供批准/拒绝选项** — 非管理员无此权限，不展示操作指引。

## 文档体系

```
/Volumes/dosia-enterprise/enterprise/inc-knowledge/
├── meetings/                    # 所有会议记录（全员可写）
│   └── YYYY-MM-DD-HHMM-概述名称.md

│   │   └── YYYY-MM-DD-HHMM-提交人-概述名称.md
│   └── processed/               # 已处理（管理员归档）
│       └── YYYY-MM-DD-HHMM-提交人-概述名称.md
├── foundations/                 # 基准文档（仅管理员可写，提案审批制更新）
│   ├── values.md                # 公司价值观（必读）
│   ├── frameworks/              # 通用方法论框架
│   ├── roles/                   # 各职能 SOP + 标准 + 知识
│   │   ├── 内容组/
│   │   ├── 流量运营组/
│   │   ├── 客户组/
│   │   └── ...
│   └── ai-integration/          # AI 应用相关框架
└── knowledge-base/              # 项目级知识库（可直接追加，无需审批）
    └── [职能组]/
```

> **foundations/ vs knowledge-base/ 区别**：
> - foundations/ = 跨项目可泛化的方法论和标准，提案审批后写入
> - knowledge-base/ = 项目特有的观察、数据、案例，可直接追加，不进入 foundations/

## 知识成熟度标记

- `[DRAFT]` — 首次提出
- `[VALIDATED]` — 至少 2 个场景验证
- `[CORE]` — 核心方法论，全员须知
- `[PENDING]` — 待决策
- `[CONFLICT: 日期]` — 与已有知识冲突
- `[NEEDS-REINFORCEMENT]` — 审查中发现的反复违反项
- `[PROPOSAL]` — 变更提案，待审批

## 注意事项

- 必须保留原文引用（> 格式）
- 每个观点/洞察标注提出者及其层级
- TODO 必须有可验证的完成标准
- 行为审查必须基于 foundations/ 中的明确条目，不凭空创造标准
- 改进建议必须具体到"应该怎么说/做"，含话术示例
- **foundations/ 的任何变更必须通过 PROPOSAL 审批，禁止直接写入**
- knowledge-base/ 可直接追加（事实性内容、案例、观察）
- 头脑风暴/讨论会侧重提取想法，不做行为审查
- 文件名必须遵循命名规则
